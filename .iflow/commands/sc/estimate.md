# /sc:estimate - 开发估算

## 触发条件
- 开发计划需要时间、工作量或复杂性估算
- 项目范围界定和资源分配决策
- 功能分解需要系统化的估算方法
- 风险评估和置信区间分析需求

## 使用方法
```
/sc:estimate [目标] [--type time|effort|complexity] [--unit hours|days|weeks] [--breakdown]
```

## 行为流程
1. **分析**: 检查范围、复杂性因素、依赖项和框架模式
2. **计算**: 应用估算方法，结合历史基准和复杂性评分
3. **验证**: 将估算与项目模式和领域专业知识进行交叉引用
4. **呈现**: 提供详细分解，包括置信区间和风险评估
5. **跟踪**: 记录估算准确性，以持续改进方法

关键行为：
- 基于估算范围的多角色协调（架构师、性能、项目经理）
- Sequential MCP集成，用于系统化分析和复杂性评估
- Context7 MCP集成，用于框架特定模式和历史基准
- 具有置信区间和风险因素的智能分解分析

## MCP集成
- **Sequential MCP**: 复杂多步估算分析和系统化复杂性评估
- **Context7 MCP**: 框架特定估算模式和历史基准数据
- **角色协调**: 架构师（设计复杂性）、性能（优化工作量）、项目经理（时间表）

## 工具协调
- **Read/Grep/Glob**: 代码库分析，用于复杂性评估和范围评估
- **TodoWrite**: 估算分解和复杂估算工作流的进度跟踪
- **Task**: 需要系统化协调的多领域估算的委托
- **Bash**: 项目分析和依赖评估，用于准确的复杂性评分

## 关键模式
- **范围分析**: 项目需求 → 复杂性因素 → 框架模式 → 风险评估
- **估算方法**: 基于时间 → 基于工作量 → 基于复杂性 → 基于成本的方法
- **多领域评估**: 架构复杂性 → 性能要求 → 项目时间表
- **验证框架**: 历史基准 → 交叉验证 → 置信区间 → 准确性跟踪

## 示例

### 功能开发估算
```
/sc:estimate "用户认证系统" --type time --unit days --breakdown
# 系统化分析：数据库设计（2天）+ 后端API（3天）+ 前端UI（2天）+ 测试（1天）
# 总计：8天，置信区间85%
```

### 项目复杂性评估
```
/sc:estimate "将单体迁移到微服务" --type complexity --breakdown
# 具有风险因素和依赖映射的架构复杂性分析
# 多角色协调，进行全面评估
```

### 性能优化工作量
```
/sc:estimate "优化应用程序性能" --type effort --unit hours
# 性能角色分析，结合基准比较
# 按优化类别和预期影响进行工作量分解
```

## 边界限制

**将会执行:**
- 提供系统化的开发估算，包括置信区间和风险评估
- 应用多角色协调，进行全面的复杂性分析
- 生成具有历史基准比较的详细分解分析

**不会执行:**
- 在没有适当范围分析和验证的情况下保证估算准确性
- 在没有适当领域专业知识和复杂性评估的情况下提供估算
- 在没有明确理由和分析的情况下覆盖历史基准
